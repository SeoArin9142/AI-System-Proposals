# 사내 AI 구축안 비교 검토안

사내 AI 개발지원 시스템 구축안을 **임원 / 대표 / 실무자가 README만 읽고도 방향을 파악할 수 있게** 정리한 저장소입니다.
현재 비교 대상은 아래 세 가지입니다.

- **완전 폐쇄형 AI 구축**
- **외부망 연계형 AI 구축**
- **하이브리드형 AI 구축**

세 안 모두 Owner / Reviewer / Orchestrator 구조를 만들 수 있지만, 보안 조건, 운영 부담, 비용 구조가 다릅니다.

## 한눈에 보기

| 구축안 | 한 줄 설명 | 언제 맞는가 | 현재 판단 |
|---|---|---|---|
| **완전 폐쇄형** | 사내 GPU 서버 + 오픈웨이트 모델 + 내부 RAG | 외부 반출이 절대 불가하고 장기 자립이 중요할 때 | **중장기 목표안** |
| **외부망 연계형** | Azure OpenAI / Bedrock / Vertex 같은 엔터프라이즈 API 중심 | 외부 전송이 정책상 허용되고 가장 빠른 도입이 중요할 때 | **조건부 검토안** |
| **하이브리드형** | 내부 Orchestrator / Git / Jenkins 유지 + 일부 AI 역할만 외부 사용 | 보안, 속도, 비용을 절충해야 할 때 | **단기 권장안** |

## 현재 결론

- **지금 바로 시작할 1차 도입안**: **보수형 하이브리드**
  - 내부: Orchestrator / Git / Jenkins / 감사로그
  - 외부: Reviewer 우선, 필요 시 제한적 Owner
  - 전송: 익명화 finding, 축약 diff, 비기밀 데이터 중심
- **중장기적으로 지향할 구조**: **완전 폐쇄형**
  - 내부 GPU 서버 + 오픈웨이트 모델 + 내부 RAG
- **빠르지만 승인 이슈가 가장 큰 안**: **외부망 연계형**
  - 외부 전송을 정책상 허용할 수 있을 때만 검토

## 구축안 요약

### 1. 완전 폐쇄형

권장 상황:
- 방산/폐쇄망 환경
- 소스 외부 반출 금지
- 내부 Git/Jenkins 재사용 가능
- 내부망에서만 코드/로그/문서 처리 필요

핵심 구조:
- 내부 LLM 서버
- 내부 Git / Jenkins
- Owner AI / Reviewer AI / Orchestrator
- 팀별 또는 개인별 workspace

운영 전제:
- 완전 폐쇄망이면 Claude / GPT / Gemini 최고급 상용 모델을 그대로 쓰기 어렵습니다.
- 현실적인 출발점은 **오픈소스 로컬 모델 + RAG + 근거 기반 리뷰 흐름**입니다.

첫 번째 성공 사례 권장:
- Sparrow / MISRA / 6016F 기반 Reviewer
- 입력: 정적분석 결과
- 출력: 규정 조항, 원인 설명, 수정 가이드, 패치 초안

폐쇄망 MVP 5줄 요약:
- 범위: 저장소 1개, 팀 1개, 중간형 GPU 서버 1대
- 목표: Reviewer 중심 AI 검토 루프가 실제 개발 흐름에서 동작하는지 증명
- 기간: 6주 구축 + 실작업 10건 파일럿
- 최소 인력: 전담 2명 + 파트타임 2명
- 교육: 개발자/리뷰어/운영자 분리 교육 후 2주 Shadow -> Assisted 정착

### 2. 외부망 연계형

권장 상황:
- 외부 AI 서비스 사용이 정책상 허용됨
- 소스 일부 또는 작업 맥락을 외부로 보내는 것이 가능함
- 빠른 도입과 높은 모델 품질이 중요함

핵심 구조:
- 내부 Git / Jenkins
- 외부 AI 서비스(Claude / GPT / Gemini 등)
- 내부 오케스트레이터
- PR / 리뷰 / 테스트 흐름 연동

운영 전제:
- 최신 상용 모델 품질을 활용하기 쉽습니다.
- 대신 외부 전송 범위, 사용료, 서비스 의존성, 감사 정책을 관리해야 합니다.

### 3. 하이브리드형

권장 상황:
- 보안은 중요하지만 완전 폐쇄형 장비를 바로 크게 사기 어려움
- 빠른 파일럿 착수가 필요함
- 내부 통제와 외부 상용 모델 품질을 절충하고 싶음

핵심 구조:
- 내부: Orchestrator / Git / Jenkins / 감사로그 / 정책 엔진
- 외부: Reviewer 우선, 필요 시 제한적 Owner
- 전송 범위: 익명화 finding, 축약 diff, 비기밀 데이터 중심

운영 전제:
- "학습 미사용"과 "외부 전송 가능"은 다른 문제이므로 전송 허용 범위를 별도로 정의해야 합니다.
- 1차 도입안으로는 가장 현실적이지만, 장기적으로는 완전 폐쇄형 이행 여부를 별도로 판단해야 합니다.

## 3안 비교

| 항목 | 완전 폐쇄형 | 외부망 연계형 | 하이브리드형 |
|---|---|---|---|
| 보안/통제 | 매우 강함 | 외부 전송 범위 통제가 필요 | 중간, 정책 설계 중요 |
| 방산/폐쇄망 적합성 | 높음 | 낮음 또는 별도 승인 필요 | 중간, 전송 범위에 따라 달라짐 |
| 초기 구축비 | 큼 | 중간 | 중간 이하 |
| 장기 운영비 | 상대적으로 예측 가능 | 사용량 과금 누적 가능 | 장비비와 API 비용이 혼합 |
| 모델 품질 | 상용 최고급 대비 낮을 수 있음 | 최신 상용 모델 활용 가능 | 상위 Reviewer 품질 활용 가능 |
| 운영 부담 | 내부가 직접 담당 | 상대적으로 낮음 | 정책/연동 운영 부담 존재 |
| 장애 영향 통제 | 내부 통제 가능 | 외부 서비스 상태 영향 받음 | 혼합 |
| Owner/Reviewer 시스템화 | 가능 | 가능 | 가능 |
| 1차 도입 현실성 | 낮음 | 중간 | **가장 높음** |
| 현재 권장도 | 중장기 목표안 | 조건부 검토안 | **단기 권장안** |

## AI 구축 vs 고급 AI 계정 여러 개 구매

여기서 비교 대상은 아래 두 가지입니다.

- **AI 구축**: Owner / Reviewer / Orchestrator 구조를 갖춘 회사 시스템
- **고급 AI 계정 구매**: Gemini Ultra, Claude Max, ChatGPT 상위 요금제 같은 개인용 고급 계정을 여러 명이 각각 사용하는 방식

핵심은 이 둘이 **같은 종류의 해법이 아니라는 점**입니다.

### 차이점 요약

| 항목 | AI 구축 | 고급 AI 계정 여러 개 구매 |
|---|---|---|
| 기본 목적 | 회사 시스템 구축 | 개인 생산성 향상 |
| 사용 주체 | 조직 공용 워크플로우 | 개별 사용자 |
| Owner / Reviewer / Orchestrator | 가능 | 사실상 없음 |
| Git / Jenkins / PR 연동 | 가능 | 제한적 또는 수동 |
| 감사 로그 / 권한 통제 | 가능 | 약함 |
| 작업공간 분리 | 가능 | 없음 |
| 지식 자산화 | 높음 | 낮음 |
| 도입 속도 | 느림 | 빠름 |
| 초기비용 | 큼 | 거의 없음 |
| 장기 통제 | 강함 | 약함 |

### 고급 AI 계정 여러 개 구매가 맞는 경우

- 당장 3~5명의 생산성을 빠르게 높이고 싶음
- 시스템 구축보다 개인 활용이 먼저임
- PR / 리뷰 / Jenkins 연동은 아직 필요 없음
- 문서 요약, 코드 설명, 초안 작성 중심으로 사용함

### AI 구축이 맞는 경우

- Owner / Reviewer처럼 역할 분리된 흐름이 필요함
- Git / Jenkins / PR / 테스트 결과까지 묶고 싶음
- 사내 규칙, 코딩 스타일, 검토 기준을 누적 자산화하고 싶음
- 사람 개인 능력에 의존하지 않는 공용 체계를 만들고 싶음

### 왜 이 차이가 중요한가

고급 AI 계정 5개를 구매하는 것은:
- **직원 5명이 좋은 AI를 각자 쓰는 것**

AI 구축은:
- **회사가 하나의 AI 작업 체계를 갖는 것**

즉,
- 계정 구매는 **도구 지급**
- AI 구축은 **업무 프로세스 구축**

입니다.

## 왜 AI 구축이 회사에 도움이 되는가

### 1. 개인 역량이 아니라 조직 자산으로 남음
- 좋은 프롬프트
- 코드 리뷰 기준
- MISRA / 6016F 같은 규정 해석 방식
- 반복적인 수정 패턴

이런 것들이 사람 개인에게만 남지 않고 시스템 안에 축적됩니다.

### 2. 같은 작업을 반복할수록 편차를 줄일 수 있음
- 사람마다 리뷰 스타일이 다름
- 사람마다 로그 해석 방식이 다름
- 사람마다 정적분석 대응 수준이 다름

AI 구축은 이 편차를 줄여서 최소한의 공통 기준을 제공할 수 있습니다.

### 3. Git / Jenkins / PR 중심으로 묶을 수 있음
- 코드 수정
- 테스트 실행
- PR 생성
- 리뷰 의견 생성
- 승인 / 반려 기록

이 흐름을 시스템화할 수 있다는 점이 단순 계정 구매와 가장 큰 차이입니다.

### 4. 회사 입장에서 관리와 감사가 쉬워짐
- 누가 어떤 작업을 했는지
- 어떤 근거로 리뷰했는지
- 어떤 결과가 생성됐는지

를 조직 차원에서 남길 수 있습니다.

### 5. 장기적으로는 사람 교체 리스크를 줄임
- 특정 사람만 알고 있는 규칙
- 특정 사람만 할 수 있는 반복 작업
- 특정 사람에게 몰린 리뷰 부담

이런 구조를 줄이는 방향으로 갈 수 있습니다.

## 장점 / 단점

### 폐쇄망 내부형 장점
- 소스 외부 유출 통제에 유리
- 방산/폐쇄망 환경에 적합
- 로그, 프롬프트, 산출물 위치를 내부에서 통제 가능
- 장기적으로 사내 자산화에 유리

### 폐쇄망 내부형 단점
- GPU 서버, 스토리지, 운영 체계가 필요
- 초기 도입비가 큼
- 모델 품질이 상용 최고급 대비 낮을 수 있음
- 모델/드라이버/운영 문제를 내부에서 해결해야 함

### 외부망 연계형 장점
- 도입 속도가 빠름
- 최신 상용 모델 품질을 즉시 활용 가능
- 초기 인프라 부담이 낮음
- 현재 상용 AI 협업 구조와 가장 유사하게 설계 가능

### 외부망 연계형 단점
- 외부 전송 허용 범위 검토 필요
- 사용량 증가 시 월 비용이 빠르게 커질 수 있음
- 서비스 정책 변화/장애 영향이 있음
- 방산/소스 외부 유출 금지 조건과 충돌 가능

## 현재 기준 권장안

### 폐쇄망이 맞는 경우
- 소스 외부 유출 금지
- 방산 또는 내부망 전용 환경
- 장기적으로 사내 전용 AI 체계를 자산화하고 싶음

### 외부망이 맞는 경우
- 빠른 도입과 높은 모델 품질이 더 중요함
- 외부 AI 사용이 허용됨
- Owner / Reviewer 시스템을 빠르게 실험해 보고 싶음

## 비용 방향

요약만 적으면:

- **폐쇄망 내부형**: 초기비용이 큼, 장기 운영 통제에 유리
- **외부망 연계형**: 초기비용은 낮거나 중간, 대신 사용료가 누적됨

정확한 수치는 아래 상세 문서를 기준으로 봅니다.
- [폐쇄망 비용/장비 상세](docs/internal-closed-network-ai-system.md)
- [외부망 비용/서비스 상세](docs/external-connected-ai-system.md)

## 결론

정리하면:

- **보안/통제 우선**이면 폐쇄망 내부형
- **속도/모델 품질 우선**이면 외부망 연계형
- **방산/소스 외부 유출 금지 조건**이면 폐쇄망이 기본 선택입니다.

다만 완전 폐쇄망에서는 현재 상용 최고급 AI와 **동일한 모델 품질**을 그대로 재현하기는 어렵습니다.  
따라서 현실적인 목표는 **오픈소스 로컬 모델 + RAG + Reviewer 우선 MVP**로 시작하는 것입니다.

## 참고

- 본 저장소는 문서 및 발표자료 공유 목적입니다.
- 원본 검토와 구현 검증은 별도 작업 저장소에서 진행되었습니다.
